MyTalli Sync Scaling Plan

API rate-limit strategy with daily sync baseline + on-demand refresh

Last updated: April 2026

Overview

A Different Scaling Axis

MyTalli has two independent scaling axes. The first is infrastructure scaling (App Service tier, SQL DTU, memory per circuit) — covered in the companion Scaling Plan document. This document covers the second axis: platform API rate limits.

Every revenue sync hits an external platform's API — and those platforms meter us on an application-wide basis, not per-user. One shared API key. One shared daily quota. Hundreds of users syncing from the same platform all eat from the same bucket.

Key insight: Etsy is the binding constraint. Stripe, Shopify, PayPal, and Gumroad all have much higher effective ceilings for our workload. When we talk about "sync scaling," we are really talking about "when do we bump into Etsy's 10K QPD quota, and what do we do about it."

Baseline cadence: once per day. Side-hustle revenue is sporadic (0-5 sales per shop per day typically) and users visit the dashboard sporadically, not continuously. Daily scheduled sync — driven server-side by ShopConnection.NextSyncDateTime — delivers the right freshness tradeoff at a fraction of the quota cost of hourly polling. There is intentionally no user-facing "Sync Now" button: manual triggers undermine the rate-limit plan and the benefit doesn't justify the complexity. If an admin debug trigger is needed later, it lives behind the Admin page.

Why Etsy hurts first: 10,000 calls/day sounds like a lot until you multiply by sync cadence and shop count. At hourly polling, just 500 shops blows past the quota. At daily polling, the same 10,000-call ceiling comfortably supports ~8,000 shops. Cadence choice is the biggest lever we have.

Scaling Phases

Three-Phase Plan

We advance one phase at a time, only when metrics force the move. Each phase has a concrete engineering response and a concrete trigger.

Phase 1 — Baseline
0 – ~8,000 Etsy shops
Daily scheduled sync + incremental delta pulls
  • Strategy: Daily baseline
  • Each shop polls once per day on schedule — driven by ShopConnection.NextSyncDateTime
  • No user-facing manual trigger (no "Sync Now" button) — undermines rate-limit plan
  • First sync is a full backfill; subsequent syncs are deltas
  • Exponential backoff on failures
  • Zero engineering cost to operate — covers the full Phase 1 range with no tuning
Phase 2 — Webhooks
~5,000 – ~15,000+ Etsy shops
Real-time push as a product differentiator
  • Strategy: Engineering
  • Wire up 4 Etsy order webhooks (order.paid, order.canceled, order.shipped, order.delivered)
  • Webhooks deliver better freshness than hourly polling at lower cost
  • Daily polling becomes a reconciliation backstop, not the primary signal
  • Market as "real-time revenue updates" — product feature, not just plumbing
  • Same pattern for Stripe, PayPal, Shopify
Phase 3 — Bespoke
Very large scale
Multi-key, dedicated workers, paid API tiers
  • Strategy: Enterprise
  • Etsy Commercial Access quota negotiation
  • Multiple API keys across isolated sub-apps where policy allows
  • Platform-dedicated sync workers with custom pacing
  • Paid-tier API access where offered
  • "A problem we'd be thrilled to have"

Previous 5-phase plan collapsed to 3. Switching the baseline from hourly to daily eliminated the need for "stretch cadence" (Phase 2) and "Commercial Access quota request" (Phase 4) as distinct phases. They still exist as tactics, but no longer get their own milestone because Phase 1 carries us ~10× further than before.

Core Math

Why These Thresholds

All numbers assume steady-state operation after first backfill. Incremental sync means 1 API call per shop per sync cycle in typical conditions (pulls only created_at >= LastSyncDateTime).

Why daily beats hourly — the scaling impact

500 shops × 24 hourly syncs × 1 call= 12,000 calls/day ✗ over 10K limit
500 shops × 1 daily sync × 1 call= 500 calls/day ✓ 5% of quota
5,000 shops × 1 daily sync × 1 call= 5,000 calls/day ✓ 50% of quota

Phase 1 at upper bound (~5,000 Etsy shops)

Scheduled daily sync (5,000 × 1 call)= 5,000 calls/day ✓ 50%
First-backfill spikes for new sign-ups≈ 300 calls/day ✓ 3%
Total load≈ 5,300 calls/day ✓ ~53% of quota

Phase 2 at 10,000 Etsy shops — webhooks take over

Webhook events (avg 0.5/shop/day × 1 fetch call)≈ 5,000 calls/day ✓
Daily reconciliation (1 call/shop, 1×/day)= 10,000 calls/day if kept
Reconciliation stretched to weekly (1/7)≈ 1,400 calls/day ✓
Webhook-dominated total~6,000-7,000 calls/day ✓

Phase 1 runway is huge. At daily baseline, the 10K QPD ceiling comfortably supports ~8,000 Etsy shops before we even think about Phase 2. A realistic user mix (30-40% connecting Etsy) means that corresponds to 20,000+ total MyTalli users — a long runway on a simple, zero-cost-to-operate architecture.

Architecture Constraint

Why We Cannot Batch Queries

A natural instinct is to ask: "Can we query data for shops A, B, and C in a single API call?" The answer is no — and it is not a quota-tier limitation we could pay around. It is architectural.

Every OAuth token is scoped to exactly one identity: one Etsy user, one Shopify shop, one Stripe account. The API has no concept of "this app is asking for data across 10 unrelated shops." Each token authenticates the app as that one user, and the endpoints return data for that one user.

Etsy

/v3/application/shops/{shop_id}/receipts — one shop per call. No multi-shop endpoint.

Stripe

Uses Stripe-Account header — one connected account per call.

PayPal

Transaction Search is merchant-scoped by token — one merchant per call.

Shopify

One OAuth install per shop — each shop is its own tenant.

Since batching is off the table, the three real call-reduction levers are:

  • Fewer, fatter queries per shop. Pull 100 records per page, not 10. Etsy allows limit=100. Shopify's GraphQL fetches orders + transactions + refunds in one query.
  • Incremental delta pulls. After backfill, query only created_at >= LastSyncDateTime. Steady-state is 0-3 records per sync.
  • Webhook ingestion. When a shop event fires, we make 1 call to fetch its detail — instead of polling "just in case." Flips the cost model from poll-count to event-count.
    • Cadence Rationale

    Why Daily, Not Hourly

    Hourly polling is the default instinct for "real-time dashboards," but it is overkill for this product. Three reasons daily is the right baseline:

    1. Side-hustle revenue is sporadic

    A typical target-market Etsy shop gets 0-5 sales per day, unevenly distributed. Most hourly syncs would return "nothing new." Those are wasted calls against a shared quota.

    2. Users visit sporadically

    Side-hustlers check their revenue dashboard weekly or a few times per week, not hourly. They do not care whether data is 1 hour old or 18 hours old — they care that it is "today."

    3. Server-side scheduling, not user-triggered

    No "Sync Now" button. A manual trigger turns rate-limit accounting from a predictable daily figure into a per-click guessing game, and the benefit (slightly fresher data in a rare edge case) doesn't justify the complexity. If an admin debug trigger is ever needed, it lives behind the Admin page.

    4. Webhooks beat hourly at freshness

    When we do need near-real-time (Phase 2), webhooks fire in seconds, not hours. Hourly polling is strictly worse than webhooks on freshness and quota cost. Skipping it entirely simplifies the architecture.

    Action Timeline

    What To Build When

    Ordered by growth milestone, not calendar date. We advance only when metrics say so.

    Now — Build Phase 1 pipeline

    Daily scheduled sync worker (polls ShopConnection.NextSyncDateTime, refreshes tokens lazily, pulls incremental data), incremental sync, exponential backoff. No user-facing manual trigger. This is the foundation every later phase builds on.

    Launch default

    At ~3,000 Etsy shops — Instrument the quota meter

    Log daily call count per platform. Dashboard or Elmah alert when Etsy count approaches 7K. Phase 1 is still comfortable here but we want the signal ready.

    Trigger: 3,000+ Etsy connections

    At ~5,000 Etsy shops — Build webhook ingestion

    Wire up Etsy's 4 order webhooks and ingest events into the sync pipeline. Daily polling continues as a reconciliation backstop. Market "real-time updates" as a product feature. Same pattern for Stripe, PayPal, Shopify.

    Trigger: Quota 70% saturated

    At very large scale — Bespoke architecture

    Evaluate Etsy Commercial Access quota negotiation, multi-key sub-apps, dedicated per-platform workers, or paid API tiers. At this scale, engineering investment is trivially affordable.

    Trigger: Webhooks + cadence exhausted
    Business Context

    Runway & Risk

    Scaling problems arrive after product-market fit, not before. By the time we cross each phase boundary, revenue has already built up to pay for the engineering or partnership work the next phase requires.

    Revenue alignment

    At 5,000 paying-tier-eligible users with 5% Pro conversion (~$3K/mo MRR), we can afford webhook engineering comfortably. Phase 1 carries us well past product-market fit before any paid engineering work is needed.

    User-mix realism

    "5,000 users" and "5,000 Etsy connections" are not the same. Most users will connect Stripe first — and Stripe has massive headroom. A realistic 30-40% Etsy mix means Phase 1 supports 15,000-20,000 total MyTalli users.

    The one real risk: a platform (most likely Etsy) unilaterally tightens rate limits, changes Commercial Access policy, or ends OAuth access. This is real but not unique — every third-party-API business carries platform risk. Our mitigation is diversification: users connecting 2-3 platforms means we are not all-in on any one platform relationship.

    Bottom line: Daily baseline + on-demand refresh is the right architecture for a side-hustle revenue dashboard. Phase 1 carries us to 15,000+ total users with zero engineering investment. Phase 2 adds webhooks as a product differentiator. Phase 3 is a "thrilled to have" problem.
