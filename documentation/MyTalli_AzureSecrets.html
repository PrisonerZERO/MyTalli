MyTalli Azure Secrets Migration

Working checklist — production secrets & configuration in Azure App Service

Last updated: May 2, 2026

Goal: Move production-correct values from appsettings.Development.json into Azure App Service so they're already in place when launch comes. Chip away over multiple sessions — no pressure to finish in one sitting.

Method: Azure Portal (not CLI). All work happens in App Service → Settings → Environment variables → App settings tab. Hierarchical config keys (with colons in appsettings.json) become double-underscore in Azure: Authentication:Google:ClientId → Authentication__Google__ClientId.

Source of truth for secret values: Source/My.Talli.Web/appsettings.Development.json on your local machine. Never commit production secrets to git or paste them into chat.

Data Protection key persistence — Azure Blob Storage

Separate task from App settings. Required so encrypted Etsy OAuth tokens (added 2026-04-26) survive App Service restarts and slot swaps.

Why this is needed: .NET's Data Protection generates a secret key at app startup and uses it to encrypt/decrypt the OAuth tokens we just added. By default that key lives in a folder on the App Service's local file system. If Azure ever moves your app to a different VM (which they do for routine maintenance), redeploys, or you slot-swap, the key can disappear — and every encrypted token in the database becomes unreadable garbage. Storing the key in Azure Blob Storage instead makes it durable.

  1. Set  Storage Account created — mytallistorage01 in MyTalli-CentralUS-ResourceGroup, Central US, Standard performance, RA-GRS replication. (2026-04-26)
  2. Set  Blob container created — dataprotection-keys, Private access level. (2026-04-26)
  3. Set  Managed identity enabled on App Service — system-assigned, Object (principal) ID fe812c6c-ea24-4f68-8952-4412393a8a3c. (2026-04-26)
  4. Set  Role assigned — Storage Blob Data Contributor granted to mytalli-web's managed identity at storage-account scope. (2026-04-26)
  5. Set  NuGet packages added — Azure.Extensions.AspNetCore.DataProtection.Blobs and Azure.Identity to My.Talli.Web. Also pinned Microsoft.Extensions.Hosting.Abstractions 10.0.* to resolve a transitive version conflict between Lamar and Azure.Core surfaced by the new packages. (2026-04-26)
  6. Set  PlatformsConfiguration.cs wired — .PersistKeysToAzureBlobStorage(blobUri, new DefaultAzureCredential()) chained onto services.AddDataProtection(), gated on DataProtection:BlobStorage:AccountName + :ContainerName being present so local dev keeps the default file-system store. Key blob path: https://{account}.blob.core.windows.net/{container}/keys.xml. (2026-04-26)
  7. Set  App Settings added — DataProtection__BlobStorage__AccountName = mytallistorage01, DataProtection__BlobStorage__ContainerName = dataprotection-keys. (2026-04-26)
  8. Pending  Deploy and verify — after first request to the deployed app, a keys.xml file should appear in the dataprotection-keys container. Will verify on next deployment.

Optional later hardening: wrap the key with an Azure Key Vault key (.ProtectKeysWithAzureKeyVault(...)) so the data-protection key itself is encrypted at rest with a customer-managed key. Skip for now — Blob alone is a big upgrade over the default.

Working approach for this task

Multi-session, no rush.

  1. Knock out App Settings in batches — group by source (OAuth, Stripe, Etsy, ACS+Unsubscribe, Connection Strings). 5-10 minutes per batch in the portal. Update the Status column from "Pending" to "Set" as you go.
  2. Save the settings after each batch (Azure won't apply until you click Save at the top of the page).
  3. Don't deploy yet — these settings will only take effect when the app actually runs in Azure. Setting them now is purely staging work.
  4. Data Protection / Blob Storage is its own pass — handle when you have a longer block of time (storage account + role assignment + code change + deploy = ~30-60 minutes).
