Overview

Architecture at a Glance

MyTalli runs as a Blazor Server application on Azure App Service. Each connected user maintains a persistent SignalR WebSocket connection (a "circuit") that holds UI state in server memory. This architecture delivers rich real-time dashboard updates but means server resources scale linearly with concurrent users.

Key insight: "Concurrent users" is not the same as "registered users." A side-hustle dashboard app typically sees 5-15% of registered users online at any given time. 1,000 registered users likely means 50-150 concurrent circuits.

Blazor Server

What Happens When a User Disconnects

1. SignalR connection breaks (tab close, sleep, network loss)
2. Blazor shows reconnection overlay to the user
3. Client auto-retries at 0s, 2s, 10s, 30s, then every 30s
4. If reconnected within 3 minutes — user resumes with no state loss
5. After 3 minutes — circuit disposed, memory freed, user must refresh

For a dashboard app, this is ideal. No user data is lost on disconnect — only UI state (which tab was selected, which filter was active). A refresh restores everything from the database.

Memory

RAM Capacity Planning

Memory is the primary scaling constraint for Blazor Server. Each circuit holds the component tree, scoped services (DbContext, repositories, adapters), and view model state.

Standard S1 (current) 1.75 GB ~1.25 GB ~500
Premium P0v3 4 GB ~3.5 GB ~1,200
Premium P1v3 8 GB ~7 GB ~2,500
Premium P2v3 16 GB ~14.5 GB ~5,000

Estimates assume ~400 KB per circuit for MyTalli (dashboard with KPI cards, charts, scoped services). ~500 MB reserved for .NET runtime, GC overhead, and OS. Actual capacity varies with usage patterns.

Scale-up trigger: When Azure portal shows memory usage consistently above 70%, scale up to the next tier. Don't wait for 90% — GC pressure degrades performance well before you hit the ceiling.

Database

SQL Database Scaling

MyTalli uses Azure SQL Database with the DTU (Database Transaction Unit) pricing model. DTUs are a blended measure of CPU, memory, and I/O.

Tier DTUs Storage Suited For
Basic (current) 5 2 GB Development, <50 concurrent users
Standard S0 10 250 GB Launch, up to ~500 concurrent users
Standard S1 20 250 GB Growing, up to ~2,000 concurrent users
Standard S2 50 250 GB Scaling, 2,000+ concurrent users

Scale-up trigger: When DTU utilization consistently exceeds 80% (check in Azure portal under SQL Database > Monitoring > DTU percentage), move to the next tier. Dashboard apps are read-heavy with simple queries — 10 DTU goes a long way.

DbContext Thread Safety

Blazor Server's parallel component rendering can cause concurrent access to the shared scoped DbContext. MyTalli handles this with a SemaphoreSlim lock on TalliDbContext that serializes all repository operations per circuit. This is a once-and-for-all fix — every new entity, page, or adapter is automatically protected through the repository layer.

SignalR

WebSocket Connection Strategy

Every Blazor Server user maintains a persistent SignalR WebSocket connection. Azure App Service handles these natively — no additional service is needed for a single instance.

When Is Azure SignalR Service Needed?

Single instance (current and recommended through Stage 3): Built-in SignalR is sufficient. App Service handles WebSocket connections natively with no connection limits beyond memory and CPU.

Multi-instance (Stage 4+): If you scale out to multiple App Service instances, you need either:
• Azure SignalR Service ($48/mo per unit, 1,000 connections/unit) — manages connection routing across instances
• Sticky sessions (ARR affinity) — routes each user to the same instance. Simpler but less resilient to instance failures.

Bottom line: You don't need Azure SignalR Service until you run multiple App Service instances. A single P1v3 (8 GB) can handle ~2,500 concurrent users — that's likely 20,000+ registered users. You'll be well into revenue before this becomes relevant.

Action Plan

Scaling Timeline

Concrete actions to take at each growth milestone.

Immediate — Switch S1 to P0v3

For ~$4/month more, get 4 GB RAM instead of 1.75 GB. This is the single highest-value optimization. Takes 5 minutes in Azure portal — no code changes, no downtime (swap slots).

Trigger: Do this now

Immediate — Upgrade SQL to Standard S0

Move from Basic (5 DTU) to Standard S0 (10 DTU). Double the capacity, 250 GB storage, and get geo-restore backup. This removes the database as a potential bottleneck during launch.

Trigger: Do this before going live

At 500 Concurrent Users — Upgrade to P1v3

When memory usage trends above 70%. Doubles CPU and RAM. Still single instance, no architectural changes needed.

Trigger: Memory consistently above 70%

At 500 Concurrent Users — Add Application Insights

At this scale, you need real observability — request traces, dependency tracking, circuit lifecycle monitoring, and alerting. Elmah covers error logging but doesn't give you performance telemetry.

Trigger: When you need to diagnose performance, not just errors

At 2,000 Concurrent Users — Evaluate Multi-Instance

Decide between vertical scaling (P2v3, 16 GB) or horizontal scaling (2x P1v3 with Azure SignalR Service or sticky sessions). Vertical is simpler; horizontal is more resilient.

Trigger: Single instance at sustained 70%+ CPU or memory

At 5,000+ Concurrent Users — Architecture Review

At this scale, consider whether Blazor Server is still the right fit or if a hybrid approach (Blazor WASM for read-heavy views, Server for real-time) would reduce server load. Also evaluate Azure SQL elastic pools or premium tier.

Trigger: Monthly infrastructure cost exceeds revenue threshold

Risk Mitigation

Production Safeguards Already in Place

DbContext Thread Safety

SemaphoreSlim on TalliDbContext serializes all repository operations per circuit. Prevents the "second operation started" exception that Blazor Server's parallel rendering can cause. Automatically protects all current and future entities.

Probe Filter Middleware

ProbeFilterMiddleware short-circuits bot/scanner traffic (.env, .php, wp-admin, expired circuit disconnects) before it reaches Blazor routing or Elmah. Reduces noise and wasted resources.

Static Error Pages

Error pages render as static SSR (no @rendermode) so they work even when the SignalR circuit fails. Users always see a branded error page, never a raw browser error.

Zero-Downtime Deployments

Staging + production deployment slots with swap. New code deploys to staging, warms up, then swaps to production with no user interruption.
