MyTalli ShopConnection ERD

Visual reference for the new sync-target layer: app.ShopConnection + app.ShopConnectionEtsy, plus the retirement of app.SyncQueue.

Last updated: April 19, 2026 · Migrations: AddShopConnectionsAndDropSyncQueue + AddShopConnectionIdToRevenueExpensePayout + MoveTokensToShopConnection

What's Changing

The previous schema was 1-to-1: one PlatformConnection per user per platform, one SyncQueue row per user per platform. That works for Stripe, Gumroad, PayPal, and Shopify, but Etsy needs a 1:N shape because successful Etsy sellers routinely run multiple shops — and Etsy caps each login at one shop, so multiple shops means multiple logins, each with its own OAuth grant and tokens.

The schema evolved across three migrations:

  • app.ShopConnection — the thing we sync from. One row per shop. Common fields live here, including the sync-queue fields absorbed from SyncQueue.
  • app.ShopConnectionEtsy — provider-specific 1-to-1 subtable (shared PK). Only Etsy gets one on day one; other platforms are always 1:1 with their PlatformConnection, so they don't need a subtable yet.
  • Tokens moved from PlatformConnection to ShopConnection (MoveTokensToShopConnection migration, April 19). Each Etsy login has its own tokens, so AccessToken / RefreshToken / TokenExpiryDateTime / PlatformAccountId belong on the shop row, not the platform row.

app.SyncQueue was dropped in the first migration — every field on it has a home on ShopConnection.

Shape of the relationship: User 1:N PlatformConnection 1:N ShopConnection 1:1 ShopConnectionEtsy. For Stripe / Gumroad / PayPal / Shopify, the middle cardinality is functionally 1:1 — those platforms will always have exactly one ShopConnection per PlatformConnection. Etsy is the outlier.

Tokens belong to the shop. PlatformConnection is now a bookkeeping row (Id, UserId, Platform, ConnectionStatus). OAuth credentials live on ShopConnection so a single MyTalli user can own N Etsy shops via N Etsy logins — each shop carries its login's tokens independently.

app.SyncQueue — Being Dropped REMOVED

The current schema has a 1:1 relationship between PlatformConnection and SyncQueue — one queue row per user per platform. Introducing ShopConnection breaks that model for Etsy: one Etsy PlatformConnection can point at multiple shops, and each shop is its own sync target with its own cadence. Leaving SyncQueue in place would require us to either (a) add ShopConnectionId to SyncQueue and leave the table with only two useful columns, or (b) maintain two parallel tables for the same concept.

Cleaner path: absorb the queue fields onto ShopConnection. One row per shop, not one row per connection.

Fields Moving to ShopConnection

  • Status → same name, same values (Pending / InProgress / Completed / Failed)
  • NextSyncDateTime → same name (default now, stepped to now + 24h after successful sync)
  • LastSyncDateTime → same name (nullable)
  • ConsecutiveFailures → same name (exponential backoff driver)
  • LastErrorMessage → same name (nvarchar(2000), nullable)
  • IsEnabled → same name (pause without disconnecting)

Fields Not Migrating

  • UserId — already on ShopConnection (denormalized)
  • Platform — reachable via ShopConnection.PlatformConnection.Platform, so it stays a single source of truth on PlatformConnection

Migration note: The Kanban lists "Schema: Drop / refactor SyncQueue" as a separate card, but doing both in one migration keeps the schema consistent end-to-end (no intermediate state where SyncQueue exists alongside ShopConnection). Proposed migration name: AddShopConnectionsAndDropSyncQueue.

How the Shape Plays Out Per Platform

Only Etsy actually needs the 1:N cardinality between PlatformConnection and ShopConnection. The table below shows how the other platforms map.

Platform Connection → ShopConnection Provider-specific subtable? Rationale
Etsy 1 : N ShopConnectionEtsy Etsy caps one shop per login, so sellers with multiple shops run multiple Etsy logins. Each login becomes its own ShopConnection row with its own OAuth tokens under the same PlatformConnection. Needs currency, URL, country, vacation mode.
Stripe 1 : 1 (always) Not yet — add when a Stripe-unique shop field appears One Stripe account per connection. No "shops" concept.
Gumroad 1 : 1 (always) Not yet No shop layer. One account = one sync target.
PayPal 1 : 1 (always) Not yet Merchant account = sync target.
Shopify 1 : 1 (always) Not yet — likely added later (ShopConnectionShopify for shop domain, currency, plan) Shop domain + currency are shop-scoped. Worth tracking per shop when we build the Shopify spoke.

Pattern: ShopConnection splits by provider only when a provider-unique shop-level field exists. Today that's only Etsy. Future subtables follow the same shared-PK convention (ShopConnectionShopify, etc.).

Revenue / Expense / Payout → ShopConnection UPDATED

A follow-up migration (AddShopConnectionIdToRevenueExpensePayout) added a nullable ShopConnectionId foreign key to each of the three downstream tables. This enables per-shop analytics (group revenue/expense/payout by shop, not just by platform) and gives dashboards a direct join to ShopConnection.ShopName for display.

Why Restrict (not Cascade)?

Shops are effectively permanent (users can pause syncing but can't disconnect). A rare hard-delete of a ShopConnection shouldn't blow away years of revenue history — Restrict forces an explicit nullout or reassignment first. It also matches the precedent set by FK_ShopConnection_User and avoids SQL Server multiple-cascade-path collisions with FK_Revenue_User.

Querying per shop: Dashboard and goal queries can now do GROUP BY ShopConnectionId (for cross-platform views) or join to ShopConnection for human-readable shop names. The existing (Platform, TransactionDate) index still serves platform-level rollups; the new FK index serves shop-scoped queries.
