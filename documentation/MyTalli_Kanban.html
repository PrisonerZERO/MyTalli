v1.0.0.8 LIVE — SQL-backed DataProtection + plaintext OAuth tokens

Shipped to https://www.mytalli.com/ on May 16 (same day as v1.0.0.7). Two-part change. Part 1 — SQL-backed DataProtection: Added Microsoft.AspNetCore.DataProtection.EntityFrameworkCore + new components.DataProtectionKey table (EF migration). TalliDbContext now implements IDataProtectionKeyContext; PlatformsConfiguration calls .PersistKeysToDbContext<TalliDbContext>(). Keys persist across container restarts / slot swaps / scale events via Azure SQL backups. Zero MI / blob / sidecar dependency — sidesteps the v1.0.0.0 deploy nightmare entirely. Part 2 — Rip out IShopTokenProtector: the at-rest encryption layer on ShopConnection.AccessToken / RefreshToken was security theater once we co-located the decryption keys in the same SQL DB as the data (zero defense against full-DB compromise) while creating a permanent operational tax (lose the keys → every connected platform requires manual reconnect). Tokens now stored plaintext; DB itself is the trust boundary. DataProtection still wraps cookies + antiforgery + the Stripe Connect onboarding cookie — those are browser-round-trip values that genuinely need encryption. Deleted: IShopTokenProtector.cs, DataProtectionShopTokenProtector.cs, ShopTokenProtectorStub.cs + the 2 now-obsolete "tokens are encrypted" tests. Stripped from 3 Domain commands + 3 Web services + 4 test files. EF Core packages bumped preview.2 → 10.0.* (stable) to satisfy the new dependency. Deploy verified end-to-end: signed in across slot swap (cookie keys persisted in SQL), first DataProtectionKey row landed on prod (key-986196af-...). 385/385 tests pass.

v1.0.0.7 LIVE — Silent data-loss sweep + sidebar highlights

Shipped to https://www.mytalli.com/ on May 16. Bundle of 5 commits queued during the Stripe Connect local end-to-end verification session and pushed together. Headline: three silent-data-corruption bugs fixed that were all variants of "code tried to be helpful with a fallback when input didn't match expectations" — see memory/feedback_fail_loud_at_boundaries.md. (1) StripeConnectSyncService + GumroadSyncService DateTime Kind: EF-loaded LastSyncDateTime arrived as Kind=Unspecified and Stripe.net's DateRangeOptions.GreaterThan serialized it as Local time, shifting the Unix-seconds filter by the local timezone offset (~5h CDT) → silently dropped every charge/payout created in that gap between syncs. Targeted fix: DateTime.SpecifyKind(value, DateTimeKind.Utc) at the SDK boundary. Project-wide follow-up: global ValueConverter<DateTime, DateTime> on TalliDbContext.OnModelCreating pins every EF-loaded DateTime to Kind=Utc across all 30+ entities — eliminates the trap forever. (2) BillingEndpoints.CreateCheckoutSession silently fell through to Pro plan checkout when ?product= wasn't in Stripe:Modules — user clicks "Add Module" for $3/mo, gets subscribed to Pro for $12/mo. Fixed: log warning + redirect to /my-plan?error=invalid_module. (3) CheckoutCompletedHandler.ResolveProductId had return 1; // fallback to Pro Monthly on any unrecognized Stripe price id — webhook silently created a Pro Monthly record instead of the real entitlement. Fixed: throw InvalidOperationException with full diagnostic context; Stripe retries, Elmah catches. Also bumped: Stripe Connect cookie expiry 10 min → 60 min (real Standard onboarding routinely takes 15-30+ min, users hit ?error=stripe_expired). UX: two-level active highlight in the sidebar — group background + active sub-link pill (--purple-soft-alt); dark-mode token fixed (was collapsed to --purple-soft, no contrast); platform parent links now go to their dashboard + properly clickable via NavLink; Manual platform's 404 dashboard link fixed (routes to /manual-entry). 5 new regression tests covering the DateTime Kind invariant + module checkout paths. 387/387 tests pass.

Module Checkout audit + 2 silent-fallback bugs fixed + 3 regression tests

The "Module Checkout extension" card was stale — the feature already shipped end-to-end in PR #43 ("ADD: Manual Entry Module & its Stripe Product Setup"). Full trace: Add Module button on /my-plan → HandleAddModule(productId) in MyPlanViewModel → /api/billing/create-checkout-session?product={id} → BillingEndpoints looks up StripeSettings.Modules → Stripe Checkout → webhook → CheckoutCompletedHandler.ResolveProductId reverse-maps price → StripeWebhookHandler creates Order + OrderItem + Subscription + SubscriptionStripe + Billing + BillingStripe. But the audit caught two silent-fallback bugs in the Web layer (same family as the Stripe Connect Kind bug — code "shipped" doesn't mean "verified"). Bug A (endpoint silent fallthrough): if ?product=3 was sent but no matching key existed in Stripe:Modules (e.g., misconfigured Azure env var), BillingEndpoints.CreateCheckoutSession silently dropped through to the Pro plan branch. User clicked Add Module for $3/mo and got subscribed to Pro for $12/mo. Fixed: now logs warning + redirects to /my-plan?error=invalid_module. Bug B (webhook silent fallback): CheckoutCompletedHandler.ResolveProductId did return 1; // fallback to Pro Monthly if the Stripe price id matched nothing in MonthlyPriceId, YearlyPriceId, or Modules. A successful Stripe charge for an unrecognized price would silently create a Pro Monthly subscription. Fixed: logs the unrecognized price + every known one and throws InvalidOperationException so the webhook fails loudly, Stripe retries, and Elmah catches it. User's payment is safe on Stripe's side until reconciled. 3 new regression tests in StripeWebhookHandlerTests: ModuleCheckout_CreatesSubscriptionWithModuleProductId, ModuleCheckout_CreatesBillingAtModulePrice, and ProUserAddsModule_CreatesSecondSubscriptionRowNotReplacingPro (proves the existing-stripe early-return only fires on duplicate of the same sub_xxx, so a Pro + module combo correctly produces two Subscription rows). SeedProductAsync helper extended with optional productTypeId. 387/387 tests pass (was 384). End-to-end manual test skipped after weighing cost vs benefit — module flow shares the same Stripe Checkout + webhook plumbing as Pro plan (which is proven in production), and the only delta (ResolveProductId mapping) is now directly unit-tested.

Stripe Connect local end-to-end verification + 2 bugs fixed + project-wide UTC converter

Full local round-trip verified against a real Stripe test-mode connected account: browser Connect Stripe → Stripe-hosted onboarding (Standard account, Jenny Rosen test data, tok_visa bank link) → /api/platforms/stripe/return → ConnectStripeCommand → ShopSyncWorker → StripeConnectSyncService → 4 rows in app.Revenue + app.RevenueStripe (gross / fee / net / balance_txn / card / risk score all correct) + 1 row in app.Payout + app.PayoutStripe (status failed, captured from the ****2227 test bank). The test surfaced two real bugs that were fixed in the same session: Bug #1 (critical, silent data loss): StripeConnectSyncService.SyncShopAsync passed shop.LastSyncDateTime (which EF Core hydrates as DateTimeKind.Unspecified) directly into the Stripe.net SDK's DateRangeOptions.GreaterThan. The SDK serializes Unspecified as Local time when building the created[gt] Unix-seconds filter, shifting the window forward by the local timezone offset (~5h CDT) and silently dropping every charge created in that gap. Targeted fix: DateTime.SpecifyKind(value, DateTimeKind.Utc) before passing to the SDK. GumroadSyncService had the same root cause via .ToUniversalTime() — bounded by Gumroad's day-precision filter but still drops sales around midnight UTC. Same fix applied. EtsySyncService was already correct (explicit SpecifyKind(Utc) in BuildMinCreatedTimestamp). Project-wide follow-up fix (same session): added a global ValueConverter<DateTime, DateTime> (+ nullable variant) in TalliDbContext.OnModelCreating that walks every entity and pins every DateTime / DateTime? property to Kind=Utc on read and normalizes Local→Utc on write. Every EF-loaded DateTime now arrives in code as UTC, full stop, across all 30+ entities — no per-property configuration. Per-service SpecifyKind(Utc) stays in place as belt-and-suspenders for in-memory DateTimes that didn't come from EF. AuditResolver unaffected (already uses DateTime.UtcNow). No display formatting, component-state, or test fallout. Bug #2 (UX): Stripe Connect cookie expiry was 10 min in PlatformEndpoints.cs — real Stripe Standard onboarding (KYC + identity + bank + 2FA + climate step) routinely takes 15-30+ min. Anyone slower than 10 min got ?error=stripe_expired and lost their session, leaving an orphaned acct_xxx in Stripe with no DB binding. Bumped to 60 min (matches Stripe's own Account Link default). We actually hit this bug live during the test — recovered the orphaned account via direct DB insert to complete the verification. 2 new regression tests: SyncShop_LastSyncUnspecifiedKind_PinnedToUtcSameWallClock in both StripeConnectSyncServiceTests + GumroadSyncServiceTests — assert the value passed to the API client is Kind=Utc with the same wall-clock value. 384/384 tests pass. Cleanup pass also nuked all dormant NextSyncDateTime=9999-12-31 fixture shops (Stripe + Etsy + Gumroad) so /platforms now shows the genuine new-user view (2 Connected, 3 Available, 0 transactions).
