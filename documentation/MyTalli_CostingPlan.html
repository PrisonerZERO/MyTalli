Growth Projections

Cost at Each Scale

How monthly costs change as the user base grows. "Concurrent users" is the key metric — typically 5-15% of registered users for a dashboard app.

How to read Azure tier names:

  • •Letter = tier family — S (Standard), P (Premium)
  • •Number = size within the family — 0 (smallest) through 3 (largest)
  • •v3 = hardware generation — version 3 is the latest
  • •S1 = Standard tier, size 1  |  P0v3 = Premium tier, size 0, version 3

Microsoft considers Standard (S-series) legacy — Premium v3 is the current recommended family.

The good news: Costs scale gradually, not exponentially. Going from 10 users to 5,000 registered users only increases the bill from $75 to ~$98/month — because P0v3 has enough headroom. The first real cost jump ($194/mo) doesn't happen until ~5,000 registered users.

Immediate Action

Optimize Before Launch

Two changes to make right now that improve capacity without significantly increasing cost.

1. Switch App Service S1 to P0v3 (+$4/mo)

Microsoft considers Standard tiers legacy. Premium P0v3 gives 4 GB RAM (vs 1.75 GB) for essentially the same price. This more than doubles your concurrent user capacity from ~500 to ~1,200. No code changes, no downtime — swap in Azure Portal.

2. Upgrade SQL Basic to Standard S0 (+$10/mo)

Basic (5 DTU) is barely adequate for production. S0 (10 DTU) doubles throughput, includes 250 GB storage (vs 2 GB), and adds geo-restore backup capability. This removes the database as a launch-day bottleneck.

Commit to a 1-year reservation on App Service for ~35% savings. Only do this once you've validated the tier is right — don't reserve before launch.

Saves ~$25-85/mo depending on tier

Azure Reserved Instances (3-Year)

3-year reservations save ~55%. Only commit this far out when you're confident in the platform choice and growth trajectory.

M

Monitor Before Scaling

Always check Azure Metrics before scaling up. If memory and CPU are under 50%, you don't need a bigger tier yet — even if user count suggests otherwise. Real usage patterns matter more than projections.

Avoids premature spending
A

Azure Cost Alerts

Set budget alerts in Azure Cost Management at 80% and 100% of your expected monthly spend. Catches unexpected spikes (runaway email sends, DDoS-driven compute) before they become a surprise bill.

Prevents bill shock

At 500+ Concurrent Users

D

Review SQL DTU vs vCore

At Standard S2 ($75/mo) and above, the vCore pricing model may be cheaper for your workload. Run the Azure DTU Calculator to compare. Dashboard apps with read-heavy, simple queries often benefit from vCore's flexibility.

Potential 10-30% savings on database
C

Cache Expensive Queries

If dashboard data is fetched frequently but changes slowly (daily/weekly revenue totals), add an in-memory cache layer. Reduces database load and may let you stay on a smaller SQL tier longer.

Delays SQL tier upgrades

At 2,000+ Concurrent Users

H

Horizontal vs Vertical Scaling

Two P1v3 instances ($248/mo + $48 SignalR = $296) vs one P2v3 ($241/mo). Vertical is cheaper and simpler. Only go horizontal when you need fault tolerance — if one instance goes down, the other keeps serving.

Choose vertical first for cost savings
S

Sticky Sessions Over Azure SignalR

If you do scale horizontally, try ARR affinity (sticky sessions) before paying for Azure SignalR Service. Sticky sessions are free and built into App Service. Azure SignalR ($48/mo per unit) is only needed if you want cross-instance resilience.

Saves $48+/mo on SignalR
Watch Out

Hidden and Often-Forgotten Costs

Email Volume at Scale

At 20,000 users with weekly summary emails enabled, that's ~80,000 emails/month ($680/mo in ACS costs alone). Monitor email opt-out rates and ensure the unsubscribe flow works well. Users who unsubscribe save you money — don't fight it.

Staging Slot Costs

Deployment slots on Standard/Premium tiers share the plan's resources (no extra charge) when idle. But if you add auto-scaling, each scaled instance duplicates all slots. At 3 instances with staging, you're paying for 6 instances.

Bandwidth / Egress

Azure charges for outbound data transfer above 5 GB/month (~$0.087/GB). Blazor Server is efficient (only sends UI diffs over SignalR), but dashboard charts and data payloads add up at scale. Unlikely to matter before 5,000+ concurrent users.

Stripe Fees

Not an Azure cost, but relevant to margin: Stripe charges 2.9% + $0.30 per transaction. On a $12/month subscription, that's $0.65/transaction (5.4% effective rate). On $99/year, it's $3.17/transaction (3.2%). Annual plans are more margin-efficient.

Multi-Shop Power Users

A minority of users connect multiple shops on the same platform (Etsy sellers with multiple storefronts, Shopify merchants with multiple stores, Stripe users with personal + business accounts). Each shop is an independent sync target, so a 3-shop user generates ~3× the sync load of a 1-shop user. Expect a modest per-user multiplier on platform API calls, background sync time, and SQL write volume at scale — plan for 1.2-1.5× vs a single-shop baseline. Platform rate limits (Etsy: 10,000 calls/day per OAuth grant, Stripe: 100 req/sec per account) cap the worst case per user. See SyncScalingPlan.html for the full per-platform sync load model.
