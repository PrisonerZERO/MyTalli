1. Overview & Common Patterns

Every revenue platform ultimately answers the same fundamental questions — but the shape, granularity, and terminology differ significantly.

Stripe

  • Amounts in cents (integers)
  • IDs are strings (ch_xxx, txn_xxx)
  • Timestamps are Unix epoch
  • Currency: 3-letter ISO lowercase
  • Pagination: cursor-based (starting_after)
  • Auth: API key + Stripe-Account header

Etsy

  • Amounts via Money object (amount / divisor)
  • IDs are integers
  • Timestamps are Unix epoch (seconds)
  • Currency: 3-letter ISO uppercase
  • Pagination: offset-based (limit + offset)
  • Auth: OAuth 2.0 (PKCE), requires commercial access

Gumroad

  • Sale amounts in cents (integers); payouts as decimal strings
  • IDs are Base64 strings
  • Timestamps are ISO 8601 strings
  • Currency: ISO lowercase on products
  • Pagination: cursor-based (page_key)
  • Auth: OAuth 2.0 (access_token param)
Key Insight: Despite different shapes, all three platforms provide: gross revenue, platform fees, net revenue, refund tracking, and payout/disbursement data. The normalized app.Revenue table can absorb all of them.

2. Stripe API

Stripe Connect (Standard or Express) gives MyTalli read access to a connected seller's account via the Stripe-Account header. Stripe has the richest, most granular financial API of the three platforms.

Primary data source: Balance Transactions — the single source of truth for all money movement. Every charge, refund, payout, fee, and dispute creates a balance transaction.

3. Etsy API

Etsy Open API v3 (REST). OAuth 2.0 with PKCE. Multi-seller apps (like MyTalli) require commercial access approval from Etsy. Required scope: transactions_r + shops_r.

Etsy Money Object: All monetary amounts use { amount, divisor, currency_code }. Divide amount by divisor to get human-readable value (e.g., { amount: 1500, divisor: 100, currency_code: "USD" } = $15.00).

4. Gumroad API

REST API at https://api.gumroad.com/v2. OAuth 2.0 with access_token parameter. Scopes needed: view_sales, view_payouts, view_profile.

ID Format: Gumroad IDs are Base64-encoded strings (e.g., "A-m3CDDC5dlrSdKZp0RFhA=="), not integers. Database columns must be NVARCHAR.

Expense / Fee Normalization Challenge

Fees are the hardest category to normalize because platforms track them very differently:

  • Stripe: Fees are per-transaction on the balance transaction itself, with a per-component breakdown
  • Etsy: Processing fees are on the Payment object, but listing fees, offsite ad fees, and other platform fees are separate ledger entries — they're not tied to a specific sale
  • Gumroad: Single gumroad_fee per sale, but affiliate commissions are separate
Decision needed: Do we store Etsy's non-sale fees (listing fees, ad fees) as app.Expense records? Or do we only normalize per-transaction fees and ignore platform overhead? This affects whether goals can track "true profit" vs. just "revenue minus processing fees."

7. Database Design Implications

Key Decisions for ERD Design

1. ID Column Types Vary by Platform

PlatformID FormatDB Column Type
StripeString with prefix (ch_xxx, txn_xxx)NVARCHAR(255)
EtsyIntegersBIGINT
GumroadBase64 stringsNVARCHAR(255)

Platform-specific ID columns on the 1-to-1 tables must accommodate these differences. The normalized app.Revenue table should store the platform transaction ID as NVARCHAR to handle all formats.

2. Money Representation

  • Stripe & Gumroad: Amounts in cents (integers)
  • Etsy: Money object with amount/divisor
  • Gumroad payouts: Decimal strings

Normalize to DECIMAL(18,2) in the database — convert cents to dollars during ingestion. Store the original platform value on the platform-specific table if needed.

3. Timestamp Formats

  • Stripe: Unix epoch (seconds)
  • Etsy: Unix epoch (seconds)
  • Gumroad: ISO 8601 strings

Normalize to DATETIME2 (UTC) in the database.

4. Currency Handling

All three platforms provide ISO currency codes. Store as NVARCHAR(3). Multi-currency support (exchange rates, presentment currency) is Stripe-specific — store on app.RevenueStripe only.

5. Platform Connection Table

Before any revenue data can be ingested, the user must connect their platform account via OAuth. This requires an app.PlatformConnection table tracking:

  • User → Platform relationship
  • OAuth tokens (access token, refresh token, expiry)
  • Platform-specific account IDs (Stripe account ID, Etsy shop ID, Gumroad user ID)
  • Connection status (active, expired, revoked)
  • Last sync timestamp

12. Expense Table Design

Platform fees that are not tied to a specific sale. Revenue already captures per-transaction fees in FeeAmount — Expense captures the rest: listing fees, ad fees, subscription fees, and other platform charges that reduce profit but aren't deducted from a sale.

Expense sits alongside Revenue in the normalized layer — parallel tables, no FK between them

Why a Separate Table?

  • Not revenue. Expenses aren't sales — they don't belong in app.Revenue, which is strictly income.
  • Different sources. Etsy charges listing fees (per item listed), ad fees (Etsy Ads), and payment processing minimums. These show up in ledger entries, not in receipt/transaction data.
  • Profit calculation. Dashboard profit = SUM(Revenue.NetAmount) - SUM(Expense.Amount). Without Expense, the dashboard overstates earnings.
  • Not all platforms have them. Stripe has no non-sale fees (everything is per-transaction). Gumroad's only fee is the flat 10% on sales. Etsy and Shopify are fee-heavy. The table handles all cases uniformly.

Data Flow

Expenses are ingested by the same sync job that pulls revenue. For Etsy, the ledger entries API returns both sales and fees — the sync job classifies each entry and routes it to Revenue or Expense accordingly. For manual expenses, users enter them directly (future feature on a dedicated page or within Manual Entry).

Relationship to Revenue

Expense has no FK to Revenue. They are parallel normalized tables — both queried by the dashboard, both filtered by platform and date range, but they represent opposite sides of the ledger. Revenue.FeeAmount handles per-sale fees (deducted from gross to get net). Expense handles standalone fees that exist outside of any sale.

13. Payout Table Design

Tracks when platforms actually send money to the user's bank account. Revenue tells you "a sale happened." Payout tells you "money arrived." The gap between the two is the platform's hold period — Stripe batches daily, Etsy holds for 3-5 business days, PayPal can hold for weeks.

Payout tracks when money actually arrives — the gap between Revenue and Payout is the platform's hold period

Why a Separate Table?

  • Cash flow visibility. A creator might have $5,000 in Revenue this month but only $3,200 in Payouts — the rest is still in transit. Without Payout, the dashboard shows earned revenue but not actual cash in hand.
  • Multi-platform timing. Stripe pays out daily, Etsy weekly, Gumroad bi-weekly. Users need to see when money from each platform will actually arrive.
  • Reconciliation. Users can compare Revenue vs Payout to spot discrepancies — holds, reserves, frozen funds, failed payouts.
  • Not a 1:1 with Revenue. One payout typically covers many sales (Stripe batches all daily sales into one payout). There's no clean FK from Payout to Revenue — it's a date-range relationship.
