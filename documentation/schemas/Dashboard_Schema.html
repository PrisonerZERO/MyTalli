MyTalli Dashboard Schema

The normalized revenue, expense, and payout data that powers the Dashboard, Goals, and Export features. All platform data (Stripe, Etsy, Gumroad, PayPal, Shopify, Manual) funnels through these tables. Provider-specific detail lives in 1:1 subtables on Revenue.

Schema: app (revenue/expense/payout/goal tables) · Siblings: auth · commerce · platform & shop

Overview

The dashboard layer is the normalization point for all financial data. The sync worker (future) or Manual Entry module writes rows here with a common shape — GrossAmount, NetAmount, FeeAmount, currency, date, platform — and the dashboard reads only from these tables. Provider subtables hold the raw detail for drill-down but never carry numbers the dashboard totals from.

  • Revenue (Revenue + RevenueEtsy / RevenueGumroad / RevenueManual / RevenueStripe) — normalized sales; provider detail in 1:1 subtables
  • Expense — platform fees not tied to a specific sale (listing fees, ad spend, subscription fees), plus manual expenses
  • Payout — platform disbursements to the user's bank (one payout covers many sales), plus manual payouts
  • Goal / GoalType — user-defined revenue targets with computed progress (no denormalized total — queries Revenue live)

Dashboard totals come from Revenue. Summary cards, charts, platform breakdowns, recent transactions — all computed from app.Revenue. Provider tables exist for drill-down, not aggregation.

No FKs between Revenue, Expense, and Payout. They're parallel. One payout covers many sales (batched by platform); a fee-only expense may have no parent sale. Joining them happens in queries, not via foreign keys.

Entity Relationship Diagram

The three transaction tables (Revenue, Expense, Payout) are parallel peers. Each has a nullable ShopConnectionId for per-shop breakdowns (null for Manual Entry). Revenue is the only one with provider-specific subtables — Expense and Payout don't have enough per-provider variation to justify them (yet).

Etsy receipt detail with currency-adjusted amounts.

Fields: ReceiptId, ListingId, ShopCurrency, AdjustedGross / AdjustedFees / AdjustedNet (nullable).

RevenueGumroad GUMROAD

Gumroad sale detail.

Fields: SaleId, LicenseKey (nullable), DiscoverFee (nullable).

RevenueManual MANUAL

User-entered sales (Manual Entry module).

Fields: Category (Sale / Service / Freelance / Consulting / Digital Product / Physical Product / Other), Quantity (default 1), Notes (nullable).

Exactly one subtable per Revenue row. The platform that produced the sale determines which subtable carries the detail. Queries drilling into a sale join to the matching subtable based on Revenue.Platform.

No provider subtables for PayPal or Shopify yet. They'll follow the same shared-PK pattern when those spokes are built. The Revenue core fields cover most needs — subtables exist only when the provider has unique detail worth preserving for drill-down.

Design Notes

Parallel tables, no FKs between them

Revenue, Expense, and Payout are three parallel lenses on the same financial reality. They're queried side-by-side by the dashboard (net revenue view), not joined via FKs. One payout covers many sales (batched by platform); a fee-only expense has no parent sale. Creating FKs would force artificial pairings.

Dedup keys

PlatformTransactionId (on Revenue / Expense) and PlatformPayoutId (on Payout — unique) are the dedup keys for sync. Manual entries synthesize a manual_{guid} value so they never collide with real platform IDs.

Restrict on ShopConnectionId

All three tables use DeleteBehavior.Restrict on the ShopConnection FK. Historical revenue/expense/payout data survives a shop being removed — the shop row itself can't be hard-deleted while any financial row references it (consistent with the "users can pause but not disconnect" rule).

Manual Entry semantics

The Manual Entry module writes Platform = "Manual" rows with ShopConnectionId = null. Per-shop breakdowns skip these rows (the "which shop?" filter is IS NOT NULL); platform breakdowns include "Manual" as a bucket alongside the real platforms.

Sample data for new users

Users with no connected platforms and no Manual Entry subscription see sample data on the dashboard (from Models/SampleData/DashboardDataset). The moment they connect a platform or activate the Manual Entry module, the flag flips and the dashboard reads from the real tables. There is never a mix of sample and real data.

Shared-PK convention on Revenue subtables

RevenueStripe / RevenueEtsy / RevenueGumroad / RevenueManual all use Revenue.Id as their own PK. C# property stays Id; DB column is RevenueId. ValueGeneratedNever(), HasForeignKey<T>(e => e.Id). Same pattern used on ShopConnectionEtsy, BillingStripe, SubscriptionStripe, and every provider auth table.
