MyTalli Authentication Schema

OAuth-only identity. A User is the core identity; each connected provider (Google, Apple, Microsoft) gets its own 1:1 subtable with a shared primary key. Adding a provider means adding a table — no new columns on existing ones.

Schema: auth · Siblings: commerce · platform & shop · dashboard

Overview

MyTalli has no local passwords. All sign-in is delegated to external OAuth providers. The schema reflects that: each provider is a peer table that references User via a shared PK — there are no provider-specific columns on the User table itself.

  • Core identity: User — one row per person, provider-agnostic
  • Provider auth tables: UserAuthenticationGoogle · UserAuthenticationApple · UserAuthenticationMicrosoft
  • Authorization: UserRole — 1:N, role names are string constants (no lookup table)
  • View: vAuthenticatedUser — unified view of (user, provider, email) across all auth providers

Provider isolation. Adding Twitch, GitHub, LinkedIn, or any other OAuth provider = new UserAuthentication{X} table with the same shared-PK pattern. Zero changes to User or any other existing table.

Email is not stored on User. It lives on each provider's auth table. The user's "primary" email is resolved via their PreferredProvider. This matters for duplicate prevention and account consolidation — see the Design Notes section below.

auth.UserRole AUTHORIZATION

Role assignments for authorization. No separate role lookup table — role names are string constants defined in Domain/Framework/Roles.cs (Admin, User). New roles are added by defining a new constant, not a new DB row.

#ColumnTypeNotes

Indexes & constraints:

  • UQ_UserRole_UserId_Role — prevents duplicate role assignments
  • IX_UserRole_UserId — per-user role lookup

Self-healing: if a user signs in with no roles, the User role is automatically assigned during sign-in.

auth.vAuthenticatedUser VIEW

A read-only SQL view that UNIONs the provider auth tables into a single (UserId, Email, DisplayName, Provider) shape. Used by the Admin user list for cross-provider queries without 3 joins.

Defined in a migration SQL script (Migrations/01_0/Views/00.auth.vAuthenticatedUser.sql). Mapped as a keyless EF entity via AuthenticatedUserConfiguration.

Not EF-managed for writes. Code queries the view via TalliDbContext directly (e.g., GetAdminUserListCommand), not through the RepositoryAdapterAsync. This is one of the rare exceptions to the adapter-only convention.

Design Notes

Duplicate prevention (same email, different provider)

When a user signs in with a new provider using an email that already exists on another provider's auth table, the app detects this and links the new provider to the existing User record — no duplicate User row is created. See EmailLookupService.

  • Detection: during sign-in, query all provider auth tables for the incoming email
  • Match found: create the new provider auth row pointing at the existing User. Update LastLoginAt.
  • No match: create a new User + provider auth row (new account)

Account consolidation (different emails, same person)

A user may have different emails on different providers (e.g., Google Gmail vs Microsoft work email). These correctly create separate User records — the app has no way to know they're the same person. Consolidation is a user-initiated action to merge two accounts they own. The schema supports it (move provider auth row to point at the kept User, migrate data, delete the orphan) but the UX flow is not yet built.

Shared-PK convention

Each provider auth table uses User.Id as its own PK — no separate identity column, no separate FK column. The C# property stays Id (so IIdentifiable works); the DB column is renamed to UserId via HasColumnName() to show data lineage. Enforced by ValueGeneratedNever() + HasForeignKey<T>(e => e.Id).

Adding a new OAuth provider

  1. Create Domain.Entities/Entities/UserAuthentication{X}.cs inheriting DefaultEntity
  2. Create the matching Domain.Data.EntityFramework/Configurations/Auth/UserAuthentication{X}Configuration.cs with shared-PK setup
  3. Add the provider to Domain/Handlers/Authentication/{X}/ (payload + handler)
  4. Wire up the OAuth middleware in Configuration/AuthenticationConfiguration.cs
  5. Register the provider in the vAuthenticatedUser UNION (update the view SQL)

No schema changes to any existing table.

No local passwords

MyTalli does not store or manage passwords. There is no PasswordHash, PasswordSalt, MfaSecret, or similar column on User. If we ever add a first-party password fallback, it would be a new UserAuthenticationLocal table following the same 1:1 shared-PK pattern.
