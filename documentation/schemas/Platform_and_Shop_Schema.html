MyTalli Platform & Shop Schema

The integration surface where MyTalli connects to external revenue platforms. A user connects a platform once (PlatformConnection), and each shop under that platform becomes a sync target (ShopConnection) with its own OAuth tokens. Etsy gets a provider subtable (ShopConnectionEtsy); other platforms add one when they need shop-level provider-unique fields.

Schema: app (platform/shop tables) · Siblings: auth · commerce · dashboard

Overview

The platform/shop layer answers two questions:

  • "Has this user connected this platform?"  → app.PlatformConnection (one row per user per platform)
  • "Which shops do we sync, how, and with what credentials?"  → app.ShopConnection (one row per shop, carrying its own OAuth tokens)

Most platforms are 1:1 between connection and shop (Stripe, Gumroad, PayPal, Shopify — one login = one account = one sync target). Etsy is the outlier: Etsy caps each login at one shop, so a seller with multiple shops runs multiple Etsy logins, and each login becomes a separate ShopConnection row with its own tokens under the same PlatformConnection.

Shape: User 1:N PlatformConnection 1:N ShopConnection 1:1 ShopConnection{Provider}. The middle cardinality is functionally 1:1 for Stripe/Gumroad/PayPal/Shopify, genuinely 1:N for Etsy.

Tokens belong to the shop, not the platform. This is what lets one MyTalli user own multiple Etsy shops under multiple Etsy logins — each shop carries its own login's OAuth credentials independently. PlatformConnection is a bookkeeping row; it does not store tokens.

How the Shape Plays Out Per Platform

Only Etsy actually exercises the 1:N cardinality between PlatformConnection and ShopConnection. The cardinality is still "many" for everything else — it's just always 1 in practice.

Etsy

1 : N

Etsy caps each login at one shop. Multi-shop sellers run multiple Etsy logins; each login becomes its own ShopConnection row with its own tokens. ShopConnectionEtsy subtable carries country, currency, URL, vacation mode.

Stripe

1 : 1 (always)

One Stripe account per connection. No "shops" concept. Subtable added when a Stripe-unique shop-level field appears.

Gumroad

1 : 1 (always)

No shop layer. One account = one sync target. No subtable yet.

PayPal

1 : 1 (always)

Merchant account = sync target. No subtable yet.

Shopify

1 : 1 (always)

One OAuth install per shop — each shop is its own tenant. Likely gets a ShopConnectionShopify subtable (shop domain, currency, plan) when the Shopify spoke is built.

Pattern: ShopConnection splits by provider only when a provider-unique shop-level field exists. Today that's only Etsy. Future subtables follow the same shared-PK convention (ShopConnectionShopify, etc.).

Design Notes

Tokens belong to the shop

OAuth credentials (AccessToken / RefreshToken / TokenExpiryDateTime / PlatformAccountId) live on ShopConnection, not PlatformConnection. On Etsy, each login has its own tokens and owns one shop — so tokens are inherently per-shop. For 1:1 platforms, the tokens still live with the shop row; there's just always one.

Per-shop Pause/Resume

The Platforms page shows a per-shop Pause / Resume toggle that flips IsEnabled. There is no "Sync Now" button — syncing is driven server-side by NextSyncDateTime; a manual trigger would undermine the rate-limit plan (see SyncScalingPlan.html).

Connect-another-shop flow (Etsy)

The "Connect another shop" button is shown for every platform — nothing technical stops a user from connecting multiple accounts on any platform. Visibility is gated only by the free-tier 1-shop-per-platform cap (PlatformItem.CanAddAnotherShop = IsProSubscriber || Shops.Count == 0). Etsy's callback distinguishes three outcomes via a ConnectEtsyResult:

  • connected — first-time connection for this user+platform
  • added — existing connection, at least one new ShopConnection row inserted
  • refreshed — existing connection, only existing shops refreshed (user authorized the same Etsy account again)

The refreshed toast is the teachable moment — it explains to the user how to actually add a different shop (sign out of etsy.com first, because Etsy's OAuth has no account switcher).

Shared-PK convention

ShopConnectionEtsy uses ShopConnection.Id as its own PK. C# property stays Id; DB column is renamed to ShopConnectionId via HasColumnName(). Enforced by ValueGeneratedNever() + HasForeignKey<T>(e => e.Id).

Sync worker (not yet built)

The schema is ready for a background worker that:

  1. Polls ShopConnection WHERE NextSyncDateTime <= now AND IsEnabled = true AND Status != 'InProgress'
  2. Refreshes the access token if TokenExpiryDateTime has passed (using RefreshToken)
  3. Calls the platform API with this row's tokens — fetches receipts/transactions/payments for this shop only
  4. Writes normalized rows to app.Revenue / app.Expense / app.Payout stamped with this ShopConnectionId
  5. Updates LastSyncDateTime, steps NextSyncDateTime to now + 24h, clears ConsecutiveFailures

Multi-shop comes out of the box — each row is self-contained.
