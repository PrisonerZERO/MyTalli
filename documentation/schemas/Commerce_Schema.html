MyTalli Commerce Schema

Products, orders, billing, and subscriptions — the payment backbone. A user's purchase flows through Product → Order/OrderItem → Billing → Subscription, with Stripe-specific data in 1:1 subtables.

Schema: commerce · Siblings: auth · platform & shop · dashboard

Overview

The commerce schema separates catalog (what's for sale), purchase (what a user bought), and billing (the money flowing for it). Subscriptions sit alongside — they represent the ongoing state of a purchase, not a separate product concept.

  • Catalog: ProductVendor · ProductType · Product
  • Purchase: Order · OrderItem (junction to Product)
  • Money movement: Billing (with BillingStripe 1:1 subtable)
  • Ongoing state: Subscription (with SubscriptionStripe 1:1 subtable)

Orders are the backbone. Subscriptions, modules, and any future products all flow through the same Order → OrderItem pipeline. A subscription is just a product.

Cascade restrictions: FK_Billing_User, FK_Subscription_User, and FK_Subscription_Product use DeleteBehavior.Restrict to avoid SQL Server's multiple-cascade-path errors. These entities remain reachable via indirect cascade paths (e.g., User → Order → Billing).

Design Notes

Shared-PK convention

Provider-specific 1:1 subtables (BillingStripe, SubscriptionStripe) use the parent's primary key as their own. No separate identity column, no separate FK column — Id serves as both. The C# property stays Id (so IIdentifiable and the repository chain work unchanged), but the database column is renamed via HasColumnName() to show data lineage (BillingId, SubscriptionId).

Cascade-delete restrictions

FK_Billing_User, FK_Subscription_User, and FK_Subscription_Product use DeleteBehavior.Restrict to avoid SQL Server's multiple-cascade-path errors. The affected entities are still reachable via indirect cascade paths (e.g., User → Order → Billing).

Adding a new billing provider

Create a sibling subtable (e.g., BillingBraintree) with the same shared-PK pattern. No schema changes to Billing. Same for new subscription providers (SubscriptionPaddle, etc.).

Audit fields

Every entity in this schema inherits from DefaultEntity: IsDeleted, IsVisible, CreateByUserId, CreatedOnDateTime, UpdatedByUserId, UpdatedOnDate. Column ordering follows the project convention: PK (0) → FK columns alphabetical → domain columns alphabetical → IsDeleted → IsVisible → audit.
