MyTalli Worker Processes

How background work runs — two workers, in-process, paced.

Last updated: April 2026

MyTalli runs two background workers inside the Blazor web app process. They keep platform OAuth tokens alive and pull revenue data from connected shops on a daily cadence. This doc captures the decisions behind that shape — what the workers do, why they aren’t two dozen of them, and why pacing is non-negotiable.

The Workers

Two workers, different jobs

TokenRefreshWorker every 6h

Keeps OAuth refresh tokens alive. Etsy’s refresh token expires in 90 days; if a user doesn’t sync in that window, the connection dies. This worker proactively refreshes any shop whose refresh token is less than 30 days from expiry. Each refresh rotates the token and resets the 90-day clock.

ShopSyncWorker every 5 min, daily per shop

Pulls new receipts from connected shops into app.Revenue and app.RevenueEtsy. Loops every 5 minutes, but each shop only syncs once every 24 hours. If access tokens are expired, refreshes inline before the call. On failure, records the error and backs off exponentially.

Decision

Why not “sync everyone simultaneously”?

The first instinct is to fan out — one task per connected shop, go. That would break the app in four separate ways, any one of which forces the same answer: pace.

1

App-wide rate limits

Etsy’s 10 QPS and 10,000 QPD are shared across all of MyTalli. Fan-out blows the limit in a single second; the whole daily quota can burn in one minute.

2

Single Azure instance

One process, one thread pool. Spawning hundreds of concurrent HTTP calls + DB writes starves HTTP threads and can OOM the box.

3

DB contention

Concurrent bulk writes to app.Revenue indexes cause lock waits. The web app, hitting the same tables, gets slower for every user.

4

Failure blast radius

If Etsy’s API hiccups, a paced worker stalls one shop. An unpaced worker fails N shops in lockstep — all backing off to the same retry moment, causing a thundering herd on recovery.

Decision

No “Sync Now” button — deliberately

Users can pause or resume a shop but cannot trigger an on-demand sync. On a shared quota, even with per-user cooldowns, manual refreshes aggregate into real quota pressure for work that almost never finds new data (side-hustle sales are sporadic — 0–5 per shop per day).

The freshness story in Phase 2 is webhooks, not a button. Etsy’s order.paid / order.shipped events deliver better freshness than any polling UI could, at lower quota cost.

Decision

Why in the web-app process (for now)

Both workers are registered via AddHostedService<T>() and run inside the same Azure App Service instance as Blazor. This trades some future flexibility for present simplicity.

Staying in-process buys us A separate worker project would buy us
One DI container, one deployment, zero setup cost Horizontal scale-out without workers fighting each other for the same shops
Worker is self-contained — cut-and-paste to a new Program.cs later Independent deploy cadence for workers vs. web
Async work is nearly free on the shared thread pool Worker survives when the web app cold-starts, restarts, or sleeps

When to extract:

  • Web app scales to 2+ instances — must extract, or add a SQL-based distributed lock (sp_getapplock), or both workers race
  • Worker takes >1s of CPU per burst — visible request latency spikes
  • Need to redeploy workers without touching web

Azure gotcha: App Service Always On must be enabled. Without it, the process is paged out after ~20 minutes of idle web traffic and the workers stop running.

Concern

Do workers interfere with web threads?

In theory they share the .NET thread pool with the web app — so, yes, it’s possible. In practice, async I/O releases the thread while waiting on the network or database, so the workers hold zero threads 99.999% of the time.

What would actually hurt:

  • Thread.Sleep instead of await Task.Delay
  • CPU-heavy work in a tight loop (parsing huge JSON, image processing)
  • Sync I/O via .Result or .Wait()
  • Shared locks that the web path also touches

Sync workload per Etsy shop is ~50ms of CPU on ~1MB of JSON — a rounding error even at 5,000 shops spread across the day.

Rules

Pacing is non-negotiable

Four independent constraints converge on the same answer: pace every call. The numbers below are what both workers ship with today.

Lever Token refresh worker Shop sync worker
Loop interval every 6 hours every 5 minutes (scans queue)
Per-shop delay 250ms 500ms
Per-page delay — 200ms (pagination within a shop)
Success cadence rotate once the 30-day window hits NextSyncDateTime = now + 24h
Failure backoff log & retry next loop 5 min × 2N, capped at 24h
Stagger — randomize NextSyncDateTime at connect time so shops don’t bunch at midnight
Batch cap per pass no cap (typically tiny) 50 shops per pass
