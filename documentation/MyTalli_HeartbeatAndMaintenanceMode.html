2. Architecture Overview

Heartbeat infrastructure (Item E)

A new BackgroundService named AdminHealthWorker ticks every 1 minute. On each tick it:

  1. Re-reads the MM flag from app.SystemSetting; if it differs from the in-memory cache held by MaintenanceModeService, updates the cache and raises the in-process StateChanged event.
  2. Writes its own heartbeat row to app.Heartbeat with Source = "AdminHealthWorker" and LastTickAt = UTCNOW. This row is what the future Sync Health Dashboard will read to confirm the worker itself is alive.
  3. (Future consumers — token expiry warnings, sync staleness, email delivery, webhook gap, API quota — plug in here as additional checks. Not in scope for E + B.)

The other two workers (ShopSyncWorker, TokenRefreshWorker) gain one new responsibility: write a heartbeat row at the end of each loop iteration. That gives the future dashboard a single uniform "is this subsystem alive" surface.

Maintenance Mode (Item B)

A boolean flag stored in app.SystemSetting (key "MaintenanceMode", value "true"/"false"). Reaches users via four paths:

TriggerMechanismLatency
User clicks / loads / refreshesMaintenanceModeMiddleware reads cached flag from singleton, returns redirect to /maintenanceImmediate
User sitting idle on local instanceAdmin's toggle raises in-process StateChanged event; MainLayout subscription navigates to /maintenance<1s
User sitting idle on different instance (future)AdminHealthWorker on that instance re-reads DB, detects stale cache, raises local event≤60s
MM turned OFFSame in-process / heartbeat paths fire with false; MainLayout subscription navigates back to /dashboard<1s local, ≤60s cross-instance

Admin indicator

Admins bypass MM entirely. To give them confidence that all live users have actually been evicted, a MaintenanceBanner component lives in MainLayout and renders only when MM is on AND the current user is admin. It shows:

  • Status pill: "Maintenance Mode is ON" with a "Turn Off" button
  • Live count: "3 non-admin users still active" — pulses red while count > 0, turns green at 0

The count is sourced from ICircuitTracker, a singleton service that uses Blazor's CircuitHandler to track open circuits + a MainLayout session counter to identify "in-app" sessions specifically (excluding circuits parked on /maintenance which uses LandingLayout).

Why MainLayout-based counting and not page-path tracking? Blazor pages don't expose their circuit ID to component code, so per-circuit path tracking requires extra plumbing. MainLayout sessions are a perfect proxy: any user on an in-app page is rendering MainLayout; the moment they're redirected to /maintenance (which uses LandingLayout) the MainLayout instance disposes and decrements the count. Same signal, simpler implementation.

Migration plan

Single migration: AddHeartbeatAndSystemSettings. Per the project rules:

  • Inherits from DbMigrationBase, overrides UpTables() / DownTables()
  • Migration folder: Migrations/{next-version}/ with Post-Deployment Scripts/00.app.SystemSetting.seed.sql for the MM seed row
  • Migration script generated to migrations/AddHeartbeatAndSystemSettings.sql (git-ignored) for production review
  • Generated script gets the standard guard block prepended (check __EFMigrationsHistory, abort with RAISERROR if already applied)
  • Run locally with Update-Database against localhost only — never against Azure

4. Domain Layer

Entities NEW

  • Domain.Entities/Entities/SystemSetting.cs — extends DefaultEntity (standard long Id + audit fields + IsDeleted + IsVisible). Adds SettingKey (string) and SettingValue (string).
  • Domain.Entities/Entities/Heartbeat.cs — extends DefaultEntity. Adds HeartbeatSource (string), ExpectedIntervalSeconds (int), LastTickAt (datetime), Metadata (string?).

Both fit the existing GenericAuditableRepositoryAsync<T> + RepositoryAdapterAsync pipeline with no special-casing. Lookups by SettingKey / HeartbeatSource use FindAsync(predicate); the unique constraint guards against duplicates.

Models NEW

  • Domain/Models/Entity/SystemSetting.cs — inherits DefaultModel (Id, IsDeleted, IsVisible); adds SettingKey, SettingValue. No audit fields per "Entity Models" rule.
  • Domain/Models/Entity/Heartbeat.cs — inherits DefaultModel; adds HeartbeatSource, ExpectedIntervalSeconds, LastTickAt, Metadata.

Mappers NEW

  • Domain/Mappers/Entity/SystemSettingMapper.cs — implements IEntityMapper<ENTITIES.SystemSetting, MODELS.SystemSetting>
  • Domain/Mappers/Entity/HeartbeatMapper.cs — same pattern

EF Configurations NEW

  • Domain.Data.EntityFramework/Configurations/App/HeartbeatConfiguration.cs — same pattern

Commands NEW

All under Domain/CommandsAndQueries/Commands/Admin/ (new subfolder; namespace My.Talli.Domain.Commands.Admin per the "Subfolder Namespace Convention" — the CommandsAndQueries umbrella is dropped from the namespace).

5. Web Layer

Services NEW

MaintenanceModeService SINGLETON

Lives in Source/My.Talli.Web/Services/Admin/MaintenanceModeService.cs. The single in-memory source of truth for the cached MM flag.

public interface IMaintenanceModeService
{
    bool IsEnabled { get; }
    event Action<bool> StateChanged;
    Task SetEnabledAsync(bool value, long actingUserId);
    Task RefreshFromDbAsync();
}
  • Startup: hosted-service initializer reads DB once at boot to populate the cache.
  • SetEnabledAsync: writes DB via UpsertSystemSettingCommand, updates cache, raises StateChanged. Called by the admin toggle endpoint.
  • RefreshFromDbAsync: re-reads DB; if value differs from cache, updates cache + raises event. Called by AdminHealthWorker every tick.
  • Singleton because all circuits + middleware + worker share the same flag.

ICircuitTracker SINGLETON

Lives in Source/My.Talli.Web/Services/Admin/CircuitTracker.cs.

public interface ICircuitTracker
{
    int InAppNonAdminCount { get; }
    event Action CountChanged;
    void RegisterInAppSession(string sessionId, bool isAdmin);
    void UnregisterInAppSession(string sessionId);
}
  • Backed by ConcurrentDictionary<string, bool> (sessionId → isAdmin)
  • Sessions are registered by MainLayout, not by the circuit handler — see "MainLayout integration" below
  • Raises CountChanged when InAppNonAdminCount changes; banner subscribes to refresh its display

Middleware NEW

MaintenanceModeMiddleware

Lives in Source/My.Talli.Web/Middleware/MaintenanceModeMiddleware.cs. Registered via app.UseMaintenanceMode() extension.

Pipeline position: after UseAuthentication (so the User claim is populated and we can check the Admin role) but before UseAuthorization and MapRazorComponents.

Logic:

  1. If !IMaintenanceModeService.IsEnabled → pass through
  2. If User.IsInRole("Admin") → pass through
  3. If request path starts with /maintenance, /api/admin/maintenance, /_blazor, /_framework, /css, /js, or /lib → pass through
  4. Otherwise → 302 Redirect to /maintenance

Workers NEW + MODIFY

AdminHealthWorker NEW

Lives in Source/My.Talli.Web/Workers/AdminHealthWorker.cs.

  • Cadence: Task.Delay(TimeSpan.FromMinutes(1))
  • Per tick:
    1. currentUserService.Set(0L, string.Empty) (system sentinel)
    2. await maintenanceModeService.RefreshFromDbAsync()
    3. await writeHeartbeatTickCommand.ExecuteAsync("AdminHealthWorker", expectedIntervalSeconds: 60)
    4. currentUserService.Clear() in finally
  • Each tick creates a fresh IServiceProvider.CreateScope() per the existing worker pattern

ShopSyncWorker MODIFY

Add a single WriteHeartbeatTickCommand.ExecuteAsync("ShopSyncWorker", expectedIntervalSeconds: 300) call at the end of each successful loop iteration. No other changes.

TokenRefreshWorker MODIFY

Same pattern: write a heartbeat row at the end of each loop with expectedIntervalSeconds: 21600 (6h).

Pages NEW

/maintenance

Lives in Source/My.Talli.Web/Components/Pages/Maintenance.razor.

  • Uses LandingLayout (not MainLayout) — keeps the maintenance experience minimal and isolated from app chrome
  • Renders the BrandHeader swoosh per the "Page Hero Branding" rule
  • Body copy: "We're tuning things up. MyTalli will be back in a few minutes." (placeholder copy; refine in implementation)
  • Static SSR (no @rendermode) — the page itself doesn't need interactivity. The auto-recovery (when MM flips off) is handled by users still having an open SignalR circuit on a previously-rendered MainLayout elsewhere; a fresh load of /maintenance stays static. (Edge case: a user who direct-navigates to /maintenance after MM is off and stays there won't auto-leave — the middleware doesn't redirect them since MM is off. That's fine; the page can show a "Back to Dashboard" link.)
  • Page is reachable directly (no MM check) — handy for testing and for the "Back to Dashboard" link to live somewhere reasonable

Components NEW

MaintenanceBanner

Lives in Source/My.Talli.Web/Components/Shared/MaintenanceBanner.razor (+ .razor.css + ViewModel in ViewModels/Shared/MaintenanceBannerViewModel.cs).

  • Rendered inside MainLayout (above the page content)
  • Renders only when MaintenanceModeService.IsEnabled && User.IsInRole("Admin")
  • Subscribes to MaintenanceModeService.StateChanged + ICircuitTracker.CountChanged in OnInitializedAsync; unsubscribes in Dispose
  • Calls InvokeAsync(StateHasChanged) on event so cross-circuit updates render correctly
  • Visual: persistent red bar at top with status text + count + "Turn Off" button. Pulses red while count > 0; turns green at 0.

MainLayout integration MODIFY

Two small additions to MainLayout.razor.cs:

  • Implements IDisposable if it doesn't already
  • OnInitializedAsync: generate a session ID, resolve User.IsInRole("Admin"), call circuitTracker.RegisterInAppSession(sessionId, isAdmin)
  • Dispose: circuitTracker.UnregisterInAppSession(sessionId)
  • Renders <MaintenanceBanner /> at the top of the layout content
  • Subscribes to MaintenanceModeService.StateChanged; on event with true AND !isAdmin, calls NavigationManager.NavigateTo("/maintenance", forceLoad: true); on event with false, navigates to /dashboard

Endpoints NEW

Add to existing Source/My.Talli.Web/Endpoints/AdminEndpoints.cs:

  • POST /api/admin/maintenance/on — admin-role-gated; calls MaintenanceModeService.SetEnabledAsync(true, currentUserId); returns 200
  • POST /api/admin/maintenance/off — admin-role-gated; calls MaintenanceModeService.SetEnabledAsync(false, currentUserId); returns 200
  • GET /api/admin/maintenance/status — admin-role-gated; returns { isEnabled, inAppNonAdminCount } (used by the banner if we need a fallback to polling, but in practice the in-process events handle it)

7. Admin UI

The existing /admin page has two sections (Single User Resend, Bulk Welcome). MM is unrelated to email; lumping it as a third section makes the page cluttered and forces users to scroll past unrelated content to reach the toggle.

Recommendation: spoke tabs on /admin

Add the project's standard spoke tabs pattern (see "Spoke Tabs" rule in CLAUDE.md) to the /admin page:

TabContent
Email (default)Existing Single User Resend + Bulk Welcome sections (unchanged)
Maintenance NEWStatus pill (ON/OFF), live count of non-admin in-app users, "Turn On" / "Turn Off" buttons, last-toggled timestamp + by-whom (from audit fields)
Sync Health (reserved — not built in this scope)Will surface heartbeat rows + per-platform status when Item C ships

Tab styling, active state, ViewModel pattern (ActiveTab string, SelectTab(string) method, @if (ActiveTab == "...") blocks with role="tabpanel") all per the existing rule. Default tab: "email" (preserves existing behavior for muscle memory).

Hero stats update for the admin hero swoosh: when MM is on, surface the count there too (e.g., "Maintenance: ON · 3 active users") so it's visible regardless of which tab is selected.

ConfirmDialog on toggle

Both "Turn On" and "Turn Off" trigger the existing ConfirmDialog component (Yes/No, danger variant) before firing the API call. Avoids accidental kills.

10. Implementation Order

Doable as one commit or several. Recommended split into 4 commits for cleaner review:

  1. Commit 1 — Schema + Domain: migration, entities, models, mappers, EF configs, 4 commands, mapper/command DI registration, unit tests for commands. No behavior change in the running app yet.
  2. Commit 2 — MM Service + Middleware + Page: MaintenanceModeService (singleton), startup initializer, MaintenanceModeMiddleware, /maintenance page, admin endpoints. Tests for service + middleware. End of this commit: an admin running SQL UPDATE app.SystemSetting SET SettingValue='true' WHERE SettingKey='MaintenanceMode' would lock out non-admin users on next request — but no UI yet to toggle from the app.
  3. Commit 3 — Heartbeat worker + worker heartbeat ticks: AdminHealthWorker, worker heartbeat ticks added to ShopSyncWorker and TokenRefreshWorker, tests. End of this commit: 3 workers ticking, MM cross-instance propagation works.
  4. Commit 4 — Indicator + Admin UI: ICircuitTracker + MainLayout session registration, MaintenanceBanner component, spoke tabs on /admin, Maintenance tab UI with toggle + ConfirmDialog. End of this commit: full happy path, admin can toggle MM from the UI and watch the count drop.

After all four ship: update CLAUDE.md per §9, increment <Version> revision in My.Talli.Web.csproj, manual smoke test (toggle MM, watch a second browser get redirected, toggle off, watch it come back).

11. Out of Scope

The following are explicitly NOT in this E + B scope. They appear here so we don't accidentally pull them in.

  • Other heartbeat consumers — token expiry warnings, sync staleness alerts, email delivery health, Stripe webhook gap detection, Etsy API quota tracking. The infrastructure (app.Heartbeat + AdminHealthWorker) supports all of these, but each is its own future work item gated on real-user data.
  • Sync Health Dashboard (Item C) — the admin surface that reads heartbeat rows + ShopConnection health and renders per-platform tiles. Comes after E + B ship. The GetHeartbeatStatusCommand is mentioned in §4 but can be deferred until C starts; or shipped now as an unused read-only command. Recommendation: defer until C to avoid YAGNI.
  • Worker Run History (Item D) — SyncRun table for trend/throughput reporting. Distinct from Heartbeat (which is "is this alive right now") — SyncRun is "how much work did this do, per run, over time". Separate work item.
  • System Heartbeat dashboard tile — the visual surface for stale-source detection. Deferred to C.
  • CSV Export (Item F) — separate work stream, lives at /export, not under /admin.
  • Maintenance Mode message customization — for first ship, the /maintenance page copy is hardcoded. Future: a second SystemSetting key like "MaintenanceMessage" with a textarea on the admin tab.
  • Maintenance Mode auto-schedule — turning MM on at a specific future time. Future feature; for now it's manual toggle only.
  • Per-circuit page tracking — the more elaborate "show admin which page each user is on" view. Out of scope; MainLayout session counting gives the needed signal without it.
