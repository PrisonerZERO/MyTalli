Navigation Pattern Wireframes

Four approaches to organizing revenue, expense, and payout data across multiple platforms plus an aggregate view. Click each tab to explore the pattern.

Pattern A
Hub & Spoke
The Dashboard is the hub — an aggregate view of all platforms. Each connected platform is a spoke with its own detail page containing tabs for Revenue, Expenses, Payouts, and an Overview. The sidebar groups connected platforms under a collapsible "Platforms" section.

Manual Entry is a unique spoke: It's the only platform where the user writes data (quick-entry row, inline editing, Notes, Categories, Quantity). API platform spokes (Stripe, Etsy, Gumroad, etc.) are read-only — they show synced data with no edit capability. Manual Entry also has an Expenses tab (for recording non-sale costs like supplies, shipping labels) but no Payouts tab (no platform disbursements). The Revenue tab IS the current Manual Entry page, adapted to the spoke layout.

Strengths

  • Clean sidebar — doesn't grow with data types
  • Each platform is self-contained with its own context
  • Platform hero can use brand colors (Etsy orange, Stripe purple, etc.)
  • Natural mental model: "I want to check my Etsy sales"
  • Scales well — adding a platform = adding a sidebar link
  • Manual Entry gets its own tailored spoke with write capabilities that don't affect other spokes

Weaknesses

  • Extra click to reach a specific data type on a specific platform
  • "Show me all my expenses" requires going to each platform separately
  • Aggregate dashboard might get heavy trying to show everything
  • User must remember which platform to check for a specific transaction
Key Idea: The user thinks in terms of platforms: "How's my Etsy doing?" The dashboard answers "How's everything doing?" Cross-platform queries (all expenses) happen via Export or Dashboard filters.
Pattern B
Data-Type First
The sidebar organizes by data type — Revenue, Expenses, Payouts. Each page shows data from ALL platforms in one unified grid with platform filter pills. The user thinks "I want to see all my revenue" rather than "I want to see my Etsy revenue."

Strengths

  • "Show me all my revenue" is one click — the primary use case
  • Easy to compare revenue across platforms side-by-side
  • Filter pills make platform drill-down fast without page navigation
  • Fewer sidebar items — clean and predictable
  • Matches how tax prep works (all revenue, all expenses, all deductions)

Weaknesses

  • Loses the "how's my Etsy doing?" holistic platform view
  • No platform-specific context (sync status, connection health)
  • Platform-specific fields (Etsy listing ID, Stripe payment method) don't fit a unified grid
  • Manual Entry needs special treatment (quick-entry row, inline edit) that other platforms don't
Key Idea: The user thinks in terms of money types: "How much did I make? How much did I spend? How much was paid out?" Platforms are just a filter dimension, not a navigation destination.
Pattern C
Hybrid
Best of both worlds. The sidebar has top-level data type pages (Revenue, Expenses, Payouts) for cross-platform views AND a collapsible Platforms section for per-platform deep dives. Two entry points to the same data — one by type, one by platform.

Strengths

  • Two mental models supported: "all my revenue" AND "how's my Etsy?"
  • Power users get cross-platform views; casual users browse by platform
  • Platform section grows dynamically as platforms connect
  • Dashboard stays focused on the aggregate — detail lives elsewhere

Weaknesses

  • Most sidebar items of any pattern — potentially overwhelming
  • Same data accessible two ways could confuse: "Where did I see that transaction?"
  • Risk of building/maintaining two grid views for the same data
  • Sidebar gets long when 5+ platforms are connected
Key Idea: Maximum flexibility at the cost of a busier sidebar. The "Money" section answers data-type questions, the "Platforms" section answers platform-health questions. Works best if the two views have genuinely different content — not just the same grid with a preset filter.
Pattern D
Dashboard + Tabs Only
The most minimal navigation. The Dashboard IS the app. Everything lives on one page with tabs for Revenue, Expenses, Payouts, and Cashflow. Platform filtering within each tab. No separate pages for data types or platforms.

Strengths

  • Simplest possible navigation — fewest sidebar items
  • Everything in one place — no hunting through pages
  • Tabs + filters give full power without page navigation
  • Fastest to build — one page, not many
  • Matches "glanceable dashboard" mental model for side-hustle creators

Weaknesses

  • Dashboard gets very heavy — KPI cards + chart + tabs + grid + filters
  • No room for platform-specific features (Etsy listing details, Stripe payment methods)
  • Manual Entry's quick-entry row doesn't fit the shared Revenue tab
  • Page performance risk — loading all data types on one page
  • URL/deep-linking is harder — can't share "my Etsy expenses" as a link
Key Idea: The ultimate simplification. Great for the "glance at my numbers" use case, but struggles when users need to drill into platform-specific detail or enter data manually. Works best if MyTalli stays a read-only dashboard without data entry features.
